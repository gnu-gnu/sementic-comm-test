# JSCC 추론 최적화 교훈

## 환경
- GPU: NVIDIA GB10 (Blackwell, Compute 12.1)
- PyTorch 2.11 + CUDA 13.0
- TensorRT 10.15, torch_tensorrt 2.11
- nvidia-modelopt 0.42.0
- 모델: JSCC SemanticCommSystem (2.7M params, 5-layer CNN encoder/decoder)
- 입력: 4K 이미지 (3840x2577) → 128x128 타일 630개

## 최적화 단계별 결과

### Encoder + Decoder 총 소요시간 (630 타일)

| 단계 | 방법 | 시간 | 가속 |
|------|------|------|------|
| 0 | Baseline (타일 1개씩, FP32) | 489ms | 1x |
| 1 | 배치 처리 (전체 타일 한번에) | 124ms | 3.9x |
| 2 | 배치 + FP16 | 72ms | 6.8x |
| 3 | 배치 + BF16 | 67ms | 7.3x |
| **4** | **TensorRT FP16** | **42ms** | **11.5x** |

### 효과 없었던 것들

| 방법 | 결과 | 원인 |
|------|------|------|
| ModelOpt NVFP4 (양자화만) | 64.5ms (FP32보다 느림) | quantize/dequantize 오버헤드 |
| ModelOpt FP8 (양자화만) | 67.7ms (FP32보다 느림) | 동일 |
| TRT + NVFP4 컴파일 | 실패 | fake tensor 호환성 문제 |
| torch.compile | 인코더에서 비효과적 | 이미 단순한 CNN 구조 |

## 핵심 교훈

### 1. 가장 큰 병목은 Python 루프였다
타일을 하나씩 for문으로 GPU에 넘기는 게 전체 시간의 대부분을 차지했다.
630개 타일을 **한 번의 배치로 묶는 것만으로 3.9배** 빨라졌다.
GPU 활용률이 낮은 상태에서 precision을 낮춰봤자 의미 없다.

**교훈: GPU 최적화의 첫 단계는 precision이 아니라 배치 크기와 데이터 전송 최소화다.**

### 2. FP16이 가성비 최고
FP32 → FP16 변환은 코드 한 줄(`.half()`)로 1.8배 추가 가속.
품질 손실 없음 (PSNR/SSIM 동일).
BF16도 비슷한 성능이지만 FP16이 더 범용적.

### 3. NVFP4/FP8 양자화는 대형 모델용
- NVFP4/FP8은 **수십억 파라미터 LLM**에서 메모리 대역폭 병목을 줄이기 위한 기술
- 우리 모델(2.7M params)은 이미 GPU 메모리에 여유 있게 들어감
- 양자화/역양자화 연산 오버헤드가 실제 연산 절감보다 커서 **오히려 느려짐**
- TRT 컴파일까지 가야 하드웨어 FP4 유닛을 직접 쓸 수 있지만, 현재 CNN 양자화 모델의 TRT 컴파일은 호환성 문제 존재

**교훈: 양자화의 이점은 모델 크기에 비례한다. 경량 CNN에는 FP16이 최적점이다.**

### 4. TensorRT는 확실한 효과가 있다
TRT FP16 컴파일로 PyTorch FP16 대비 추가 1.7배 가속 (특히 인코더: 32ms → 12ms).
다만 컴파일 시간이 수십 초 소요되므로, 사전 컴파일 후 엔진을 저장해두는 게 실전 운용에 적합.

### 5. 최적화 우선순위
```
① 배치 처리 (가장 큰 효과, 구현 간단)
② FP16 변환 (한 줄, 품질 손실 없음)  
③ TensorRT 컴파일 (추가 가속, 컴파일 시간 필요)
④ FP8/FP4 양자화 (대형 모델에서만 유효)
```

## End-to-End 통신 속도 비교 (최적화 후)

### JSCC 인코딩/디코딩 시간
- Encode: 12ms (TRT FP16)
- Decode: 31ms (TRT FP16)
- 합계: 43ms

### JPEG vs JSCC 대역폭별 E2E

| 대역폭 | JPEG E2E | JSCC E2E | 결과 |
|--------|----------|----------|------|
| 10 Mbps | 974ms | 587ms | JSCC 1.7x 빠름 |
| 1 Mbps | 9.4초 | 5.2초 | JSCC 1.8x 빠름 |
| 100 Kbps | 93초 | 52초 | JSCC 1.8x 빠름 |
| 10 Kbps | 934초 | 516초 | JSCC 1.8x 빠름 |

대역폭이 낮아질수록 전송 시간이 지배적이 되어 JSCC의 전송량 이점이 부각된다.
